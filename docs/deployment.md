# Deployment — Render free tier (today) → VPS / AWS (future)

## Runtime shape

`self-backend` is a Spring Boot 3.2 modular monolith. Runs as one process,
binds to `$SERVER_PORT` (default 8080), reads config from environment
variables, connects to an external Supabase Postgres, and (optionally) a
Redis or Valkey instance for distributed rate-limits + refresh-token store
+ session sharing. All integrations are behind `@ConditionalOnProperty`
bean gates — missing secrets on disabled integrations do not crash boot.

Actuator surfaces at `/actuator/health`, `/actuator/health/liveness`,
`/actuator/health/readiness`, `/actuator/prometheus`, `/actuator/metrics`.
Never expose these publicly — a reverse proxy or platform ingress must
whitelist `/actuator/*` for the health checker only.

## Render free plan — what to know

- **512 MB RAM.** JVM heap runs at `MaxRAMPercentage=60` = ~307 MB. Modulith
  boots but has little headroom. Move to `starter` (2 GB) once traffic
  crosses a few RPS.
- **Sleeps after 15 min of no traffic.** Cold start on wake ≈ 30-60 s.
  First request after wake will time out — retry, or ping the URL every
  10 min from a scheduler if this hurts.
- **No persistent disk.** Anything written to `/tmp/edss` is scratch.
  All durable state must live in Supabase or the storage backend.
- **No Redis / Valkey on free tier by default.** In-memory stores take
  over. Fine for a pilot. Multi-instance and rate-limit accuracy needs
  a real cache — provision one and flip `REDIS_ENABLED=true`.

## Feature-flag matrix

Every external integration is off by default and requires **both** a flag
and its credentials to activate. Missing credentials on an off integration
are silently ignored. Turning any flag on without setting the credential
logs a warning at boot and the affected feature is effectively disabled.

| Integration          | Flag env var                | Credential env vars                                            |
| -------------------- | --------------------------- | -------------------------------------------------------------- |
| Cache (Redis/Valkey) | `REDIS_ENABLED=true`        | `REDIS_HOST`, `REDIS_PORT`, `REDIS_PASSWORD`, `CACHE_PROVIDER` |
| Sentry               | `SENTRY_ENABLED=true`       | `SENTRY_DSN`                                                   |
| Email (Resend)       | `NOTIFICATIONS_EMAIL=true`  | `RESEND_API_KEY`, `MAIL_FROM`                                  |
| Stripe               | `PAYMENTS_STRIPE_ENABLED=true` | `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`                |
| Razorpay             | `PAYMENTS_RAZORPAY_ENABLED=true` | `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`, `RAZORPAY_WEBHOOK_SECRET` |
| WhatsApp (Twilio)    | `NOTIFICATIONS_WHATSAPP=true` + `WHATSAPP_ENABLED=true` | `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_WHATSAPP_FROM` |
| Cal.com              | `CALENDAR_PROVIDER=calcom`  | `CALCOM_API_KEY`, `CALCOM_WEBHOOK_SECRET`                      |
| Calendly             | `CALENDAR_PROVIDER=calendly` | `CALENDLY_API_KEY`, `CALENDLY_WEBHOOK_SECRET`                 |
| Google OAuth         | `GOOGLE_CALENDAR_SYNC=true` | `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`, `GOOGLE_OAUTH_REDIRECT_URI` |
| Supabase Storage     | `FILE_BACKEND=supabase`     | `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`                    |
| S3 Storage           | `FILE_BACKEND=s3`           | `AWS_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_S3_BUCKET` |

Two things are **always required** because the app cannot function without
them:

- Postgres — `DB_URL`, `DB_USERNAME`, `DB_PASSWORD`.
- Auth — `JWT_SECRET`, `SECRET_ENCRYPTION_KEY` (both auto-generated by
  Render when `generateValue: true` is set in `render.yaml`).

`CORS_ALLOWED_ORIGINS` is not strictly required — leave it empty for a
same-origin deployment (single reverse proxy in front of both frontend and
backend). Set it once the browser-facing frontends land on their own
Vercel origins.

## Cache: Redis vs Valkey

They are the same wire protocol. Valkey is a BSD-licensed fork of Redis
created after the license change. The Spring Data Redis client works with
either without modification.

Pick your backend:

- **Redis:** Render Key Value add-on (up to `pro` plan), Upstash, AWS
  ElastiCache Redis, or self-hosted.
- **Valkey:** AWS ElastiCache Valkey, DigitalOcean Managed Valkey, or
  self-hosted.

Then:

```
REDIS_ENABLED=true
CACHE_PROVIDER=redis          # or `valkey` — informational only
REDIS_HOST=<endpoint>
REDIS_PORT=6379
REDIS_PASSWORD=<password>
HEALTH_READINESS_INCLUDE=db,redis,diskSpace,ping
```

The `CACHE_PROVIDER` value only surfaces in logs so operators can tell at
a glance which backend they're pointed at. Flipping it changes nothing in
code — the migration is an endpoint swap.

## First-time Render setup

1. Push this repo to GitHub if not already.
2. Render dashboard → **New +** → **Blueprint** → point at the repo, branch
   `main`. Render reads `render.yaml`, creates:
   - `self-backend` (web service, Docker runtime, plan `free`).
   - (Optional) `self-backend-redis` if you uncomment the block at the bottom.
3. Render prompts for every env var marked `sync: false`. Fill in **at
   minimum**:
   - `DB_URL`, `DB_USERNAME`, `DB_PASSWORD` (Supabase prod project).
   - Skip every other `sync: false` value on first deploy — the integration
     stays dormant.
4. Deploy. First cold start ~2 min (Flyway migrations run against Supabase
   Postgres on boot).
5. `GET /actuator/health/readiness` should 200 once ready. Verify from the
   Render Shell tab:
   ```
   curl -sS localhost:$PORT/actuator/health
   ```

## Opting integrations in later

For each integration you want to turn on (example: Sentry):

1. Render dashboard → self-backend → Environment → set
   `SENTRY_ENABLED=true` and paste `SENTRY_DSN=<real-dsn>`.
2. Save. Render redeploys automatically.
3. Verify with a test error and a 30-second wait for it to appear in
   Sentry.

Same pattern for Stripe / Razorpay / Twilio / Cal.com / Google / Supabase
Storage. The flag column of the matrix above tells you which env var to
flip.

## Release flow

`release.yml` builds and pushes the production image to GHCR after CI
passes on `main`:

- Tags: `ghcr.io/<owner-lc>/self-backend:<sha>`, `:<short12>`, `:main`,
  `:latest`.
- BuildKit cache is stored in GitHub Actions cache so subsequent pushes
  reuse Maven and Docker layer state.

Render deploys directly from the repo Dockerfile via `render.yaml`'s
`runtime: docker`, so it does not consume the GHCR image today. The image
is published anyway so a later migration to a VPS or AWS ECS is a
`docker pull` away.

## Migration path — VPS

When traffic outgrows Render (or when the monthly cost crosses the price
of a Hetzner CX21):

1. Provision the VPS with Docker + docker compose + a reverse proxy
   (Caddy or Traefik) that terminates TLS via Let's Encrypt.
2. Copy the env-var values from Render into a systemd EnvironmentFile or
   a Docker secret; do not commit.
3. `docker pull ghcr.io/<owner-lc>/self-backend:<sha>` and run under the
   reverse proxy. Health check target: `/actuator/health/readiness`.
4. Keep Supabase Postgres, Resend and Sentry — only the compute
   layer moves. If you were on Render Key Value for Redis, swap to a
   self-hosted Redis / Valkey container on the same VPS.
5. Cut DNS over after a green health check on the new instance for at
   least 24 h.

## Migration path — AWS ECS on Fargate

1. Mirror the GHCR image into ECR — either `docker pull` + `docker push`,
   or point a Copilot / CDK task at the GHCR URL directly (ECS accepts
   external registries with proper IAM).
2. Store every `sync: false` secret in AWS Secrets Manager, wire into
   the task definition.
3. Front with an ALB → target group on port 8080, health check path
   `/actuator/health/readiness`.
4. RDS Aurora Postgres (or keep Supabase) — only `DB_URL` /
   `DB_USERNAME` / `DB_PASSWORD` change.
5. ElastiCache Valkey (or Redis) replaces the Render add-on. Point
   `REDIS_HOST` etc. at the new endpoint; `CACHE_PROVIDER=valkey`
   documents the switch.

The Dockerfile does not change between these targets.

## Local docker compose

`docker-compose.yml` at the repo root brings up Postgres 14, Mailhog, and
optionally Redis (profile `scale`). Not intended for prod.

```bash
docker compose up -d postgres mailhog
cp .env.development.example .env
mvn spring-boot:run
```
